# Architecture Guardrails

This document defines the current architecture boundaries for the monorepo and the rules that should guide future implementation.

## Goals

- Keep responsibilities explicit across `web`, `api`, and `worker`.
- Avoid premature shared-package extraction.
- Protect the codebase from cross-app coupling and duplicated domain logic.
- Make later scaling decisions traceable through ADRs.

## Current Topology

The repository currently contains three runtime applications:

- `apps/web`
- `apps/api`
- `apps/worker`

There are currently no shared packages under `packages/`, which is intentional at this stage.

## Runtime Responsibilities

### Web

`apps/web` is responsible for:

- browser-facing UI
- product flows and screen composition
- design-system primitives and tokens
- calling the API through explicit HTTP boundaries

`apps/web` must not:

- connect directly to the database
- own queue-processing logic
- duplicate business rules that belong in backend services

### API

`apps/api` is responsible for:

- request validation and transport boundaries
- authentication and authorization
- domain orchestration
- persistence and transactional business logic
- webhook entrypoints and integration coordination

`apps/api` must not:

- contain UI concerns
- perform long-running rendering or queue-consumer work inline when it should be delegated to the worker
- expose storage or queue implementation details directly to the browser

### Worker

`apps/worker` is responsible for:

- queue consumption
- asynchronous rendering and artifact generation
- long-running or resource-heavy background jobs

`apps/worker` must not:

- become a second API surface
- own user-facing authentication flows
- duplicate API-owned orchestration logic unless the logic is explicitly extracted into a shared module

## Integration Rules

- `web` talks to `api` through explicit HTTP interfaces only.
- `api` and `worker` may share infrastructure dependencies, but they should not share behavior by copy-paste.
- Background work should move through queues and persisted job state, not hidden in-process side effects.
- Artifacts generated by the worker should be stored in object storage and referenced by stable identifiers.

## Shared Code Policy

Do not create a shared package just because two files look similar once.

Create a shared package only when all of the following are true:

1. The logic is reused across more than one app.
2. The shared behavior is stable enough to justify an ownership boundary.
3. Extracting it reduces duplication without making dependency flow harder to understand.

Good early candidates for shared packages later:

- domain schemas and validation contracts
- rendering data contracts
- typed client helpers
- shared UI primitives used beyond the web app shell

Bad early candidates:

- one-off helpers
- thin wrappers used by only one app
- unstable code that is still changing every sprint

## Dependency Direction

Use this dependency direction as the default:

- UI primitives depend on tokens
- screens depend on UI primitives
- API controllers depend on services
- services depend on domain logic and infrastructure adapters
- worker jobs depend on domain contracts and infrastructure adapters

Do not allow infrastructure concerns to leak upward into every layer.

## Domain Ownership

The long-term domain areas implied by the current product docs are:

- auth and user setup
- contacts and addresses
- templates and rendering
- moments and draft generation
- orders and fulfillment
- notifications
- payments

As implementation grows, keep each domain coherent. Prefer adding focused modules inside `api` or `worker` before splitting across many folders with unclear ownership.

## File and Module Conventions

- Keep app-specific logic inside its owning app.
- Prefer descriptive module names over generic names like `utils`, `helpers`, or `common`.
- Keep transport-layer code separate from domain behavior where the complexity justifies it.
- Add a short comment only when the code’s intent or constraint is not obvious from the implementation.

## Decision Record Policy

Important architectural changes should be recorded as ADRs under [`docs/adr`](./adr).

Use an ADR when changing:

- application boundaries
- deployment topology
- storage model
- queueing model
- shared-package strategy
- major framework or provider choices

## Current Architectural Priorities

1. Keep local development deterministic.
2. Keep the API as the single business boundary.
3. Keep worker behavior explicit and queue-driven.
4. Grow the web app through reusable primitives instead of page-local styling drift.
5. Delay shared-package extraction until reuse is real.
